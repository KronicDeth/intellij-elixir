Reports type names in type specifications that cannot be resolved to any type definition in scope.

In Elixir, a type name used in a @type, @typep, @opaque, @spec, @callback, or @macrocallback must resolve to a type that is defined, imported from another module, or provided as a built-in. When the type cannot be resolved, the specification is misleading and tools such as Dialyzer will not behave as intended.

This inspection flags the type name (for example, missing in @spec convert(Other.missing()) :: :ok) when it does not resolve to a type definition. For qualified names, the message includes the qualifier (e.g. Type 'Other.missing/0' is not defined); for unqualified names it reports the bare name and arity (e.g. Type 'nope/0' is not defined).

The following are recognised and never flagged:

  • Built-in types such as integer(), term(), and atom().
  • Type variables declared in a @type head, for example a and b in @type pair(a, b) :: {a, b}.
  • Type variables bound by a when guard, for example a in @spec identity(a) :: a when a: term().

Qualified type names whose qualifier module itself cannot be resolved are not flagged here; the unresolved module is reported by the Unresolvable module qualifier inspection instead.

This inspection only runs on files under Source or Test Source roots (e.g. lib/, test/). Files in unmarked directories like config/, and Elixir fragments injected into documentation, are not inspected.
